Testing Guidelines for Solarium Customer App

Overview

This document outlines testing practices, coverage requirements, and quality standards for the Solarium Customer App.

Coverage Requirements

Global Coverage Targets

  • Business Logic: ‚â•87% (store/, api/, utils/, storage/)

Testing Types

1. Unit Tests

  • Location: src/**/*.test.{ts,tsx}
  • Purpose: Test individual functions and components
  • Coverage: All public methods and critical paths

2. Integration Tests

  • Location: src/__tests__/integration/
  • Purpose: Test component interactions and workflows
  • Focus: Authentication flows, storage integration, error handling

3. Performance Tests

  • Location: src/__tests__/performance/
  • Purpose: Validate performance requirements
  • Thresholds:
    • Storage operations: <30ms
    • Concurrent operations: <50ms average
    • Memory leaks: <1MB increase per 100 operations

4. Accessibility Tests

  • Location: src/**/*Accessibility*.test.{ts,tsx}
  • Purpose: Ensure accessibility compliance
  • Tools: react-native-accessibility-engine, axe-core

Secure Storage Testing

Test Scenarios

  1. Success Path: Normal storage operations
  2. Failure Path: Handle storage errors gracefully
  3. Migration Path: Upgrade from legacy storage
  4. Performance Path: Meet timing requirements
  5. Concurrency Path: Handle simultaneous operations

  resetStorageFailure,

Quality Standards

Code Quality

  • ESLint rules enforced with zero warnings in CI
  • Prettier formatting required
  • TypeScript strict mode
  • No circular dependencies (import/no-cycle)

Security Standards

  • No hardcoded secrets or tokens
  • Secure storage for sensitive data
  • Regular dependency audits
  • Proper error handling without information leakage

Performance Standards

  • Secure storage operations <30ms
  • Bundle size monitoring
  • Memory leak prevention
  • Efficient re-renders

Manual Testing

Secure Storage Verification

Android

# Verify no plain text tokens
adb shell grep -R "demo_token" /data/data/com.solariumcustapp/shared_prefs
# Should return no matches

iOS

  1. Open Keychain Access app
  2. Look for entries under app's access group
  3. Verify token is stored as binary/Base64

Device Testing

  • Test on physical devices (not just simulators)
  • Test storage behavior across app restarts
  • Test migration from legacy versions
  • Test storage failure scenarios

Continuous Integration

Pipeline Stages

  1. Lint: ESLint + TypeScript + Prettier
  2. Test: Unit + Integration + Performance
  3. Security: Audit dependencies + vulnerability scan
  4. Quality: SonarQube analysis
  5. Build: Platform-specific builds

Quality Gates

  • All tests must pass
  • Coverage thresholds must be met
  • Zero ESLint warnings
  • SonarQube Quality Gate: A
  • Security audit: No high/critical vulnerabilities

Troubleshooting

Common Issues

Coverage Not Meeting Thresholds

  1. Check excluded files in jest.config.js
  2. Add tests for uncovered lines
  3. Verify test files are being discovered

Storage Tests Failing

  1. Check mock setup in jest.setup.js
  2. Verify SecureStorageError imports
  3. Reset storage state between tests

Performance Tests Timing Out

  1. Check for async/await issues
  2. Verify mock implementations
  3. Increase timeout for slow operations

Debug Commands

npm test -- --verbose

Best Practices

Writing Tests

  1. Follow AAA pattern (Arrange, Act, Assert)
  2. Use descriptive test names
  3. Test edge cases and error conditions
  4. Mock external dependencies
  5. Clean up after tests

Test Organization

  1. Group related tests in describe blocks
  2. Use beforeEach/afterEach for setup/cleanup
  3. Keep tests focused and independent
  4. Use test utilities for common operations

Performance Testing

  1. Use performance.now() for timing
  2. Test with realistic data sizes
  3. Monitor memory usage
  4. Test concurrent operations
  5. Set appropriate thresholds
